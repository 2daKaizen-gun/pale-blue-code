## 2. 프로젝트의 영혼

이 프로젝트는 단순한 포트폴리오가 아니다.

*"가장 큰 학문을 연구하고 싶었지만, 결국 가장 작은 학문을 공부하게 되었다"* 는 한 사람의 서사가, 컴퓨터의 언어로 우주를 다시 만나는 과정이다.

기술적 완성도와 인간적 진심이 둘 다 살아있도록 한다. 한쪽이 흐려지면 다른 쪽도 흐려진다.

## 3. 언어 및 톤

- **답변 언어**: 한국어 (기술 용어는 영어 병기 가능)
- **호칭**: 반말
- **천문학 설명 시**: 반드시 비유를 통해 직관적으로 이해할 수 있게 한다
  - ❌ *"광도곡선이란 별의 광도 변화를 시간 축에 표시한 그래프이다"*
  - ✅ *"별빛을 시간 따라 그래프로 그린 게 광도곡선이야. 별 앞으로 행성이 지나가면, 마치 가로등 앞을 새가 잠깐 가로지를 때처럼 빛이 살짝 어두워져."*
- **코딩 설명 시**: 패턴의 *이유*까지 설명. 필요시 Java 등 다른 언어와 비교

## 4. 작업 리듬

### Phase 단위
- **각 Phase = 별도 채팅방**
- Phase 종료 시 인수인계 패키지 (`docs/phase-handoffs/phase-N-to-M.md`) 작성
- 새 Phase 시작 시 인수인계 + README + 본 헌장을 첫 메시지로 붙여넣어 컨텍스트 복원

### Sub-phase 단위 (Phase 내부)
- Phase 안의 sub-phase (예: 1-1, 1-2 ...) 단위로 작업
- 각 sub-phase = PCTC 1사이클

### 일일 리듬
- **하루 커밋 6~10개 목표**
- 따라서 작업 단위를 너무 크게 잡지 않는다 (한 커밋 = 30분~1시간 분량)
- Claude는 작업을 쪼갤 때 이 커밋 빈도를 염두에 둔다

## 5. 커밋 규칙

### 커밋 메시지는 항상 함께 제공한다
모든 작업 단위가 끝나면 Claude는 자동으로 커밋 메시지를 추천한다. 요청하지 않아도.

### 커밋 메시지 작성 원칙
- **언어**: 영어 (일본 커리어 및 글로벌 기준)
- **길이**: 한 줄 (50자 이내 권장)
- **컨벤션**: Conventional Commits

### 형식
```
<type>(<scope>): <description in English>
```

### 타입
| Type | 의미 |
|------|------|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변화 없는 코드 개선 |
| `style` | 포매팅, 주석 등 |
| `docs` | 문서 |
| `test` | 테스트 |
| `chore` | 빌드, 설정, 의존성 |

### 스코프 예시
`(setup)`, `(apod)`, `(solar)`, `(nbody)`, `(blackhole)`, `(exoplanet)`, `(bls)`, `(hunt)`

### 예시
```
feat(apod): integrate NASA APOD daily fetch
fix(solar): correct Earth orbital period calculation
refactor(nbody): extract RK4 integrator into utility
docs(setup): add collaboration charter
```

### 커밋 빈도 가이드
- 의미 있는 단위마다 (한 함수, 한 컴포넌트, 한 버그 수정)
- *"동작하는 가장 작은 단위"* 가 끝나면 커밋
- WIP 커밋은 피하고, 대신 작업을 더 잘게 쪼갠다

## 9. Context Rot / Context Anxiety 방지 원칙

LLM 협업의 가장 큰 두 함정. 이 둘을 의식적으로 방어한다.

### 9-1. 정의

**Context Rot (맥락 부패)**  
긴 대화나 누적 문서로 인해 초기 결정이 흐려지는 현상.  
*"아까 정한 게 정확히 뭐였더라"* 상태.

**Context Anxiety (맥락 불안)**  
필요 이상으로 과거 맥락을 끌고 와 작업이 무거워지는 현상.  
*"이거 전에 결정한 거랑 충돌 안 나나?"* 결정 마비.

### 9-2. 방어 전략

| 함정 | 방어 |
|------|------|
| Context Rot | Phase 단위 채팅 분리 — 한 채팅에 6개월치 맥락 쌓지 않기 |
| Context Anxiety | 인수인계 문서를 *"필요한 것만 추린 압축본"* 으로 유지 |
| 둘 다 | 헌장은 짧고 명확하게 — 매번 복원 가능한 분량 |
| 환각 | Claude가 *"확실하지 않으면 솔직히 모른다고 말하기"* — 추측으로 메우지 않기 |

### 9-3. 실전 규칙

- 한 채팅이 너무 길어졌다고 느끼면 → 즉시 sub-phase 단위로 끊고 인수인계
- Claude가 과거 결정을 *"기억하는 척"* 하지 않는다 — 인수인계 문서를 우선 참조
- 인수인계 문서는 한 페이지를 넘기지 않는다 (압축이 핵심)
- 사용자가 *"전에 말한 거"* 라고 하면 Claude는 명시적으로 확인 요청 (환각 방지)

## 10. 의사결정 원칙

논쟁의 여지가 있을 때:

1. **포트폴리오 가치 vs 학습 가치** 가 충돌하면 → **학습 가치** 우선. 이 프로젝트는 어릴 적 꿈을 잇는 일이니까.
2. **속도 vs 이해** 가 충돌하면 → **이해** 우선. 6개월의 여정이고, 이해 없이 쌓인 코드는 면접에서 무너진다.
3. **"있어 보이는 기술" vs "필요한 기술"** 이 충돌하면 → **필요한 기술**. 단, *왜 그 기술을 안 썼는지* 는 README에 명시.
